# Tikitak PR Review Guide

프로젝트 컨텍스트를 근거로 Pull Request를 리뷰한다.
최종 리뷰 문구는 자연스러운 한국어로 작성하고 기술 용어는 정확하게 사용한다.

## 우선순위

1. `AGENTS.md`, `.claude/rules`, decisions 등 저장소 규칙 준수
2. 정확성, 회귀 가능성, 데이터 계약 안정성
3. 접근성과 모바일 사용자 흐름
4. 타입 안정성
5. 테스트 및 검증 누락

## Tikitak 프로젝트 규칙

- `src/app`은 앱 부트스트랩, providers, routes, layout, 전역 스타일을 담당한다.
- `src/pages`는 라우트 단위 기능을 담당하고 `src/shared`에 의존할 수 있다.
- `src/shared`는 `src/pages`에 의존하지 않는다.
- 페이지 간 직접 import를 피하고 공용 경계가 필요하면 `shared`로 승격한다.
- 서버 데이터는 TanStack Query, 클라이언트 전역 상태는 Zustand, UI 로컬 상태는 `useState`를 우선한다.
- API 호출과 요청·응답 타입은 `shared/api` 경계를 통해 관리한다.
- 비즈니스 로직 Hook과 인터랙티브 컴포넌트 변경에는 회귀 테스트 필요성을 검토한다.
- 테스트는 대상 코드 옆에 co-location한다.
- 모바일 웹과 Capacitor 환경의 내비게이션, safe-area, native bridge 회귀를 함께 검토한다.

## 리뷰 대상

- 구현이 저장소의 문서화된 규칙이나 결정과 충돌한다.
- diff가 동작 오류, 회귀, 접근성 문제, 보안 문제 또는 불안정한 타입 경계를 만든다.
- 변경된 핵심 동작에 회귀를 방지할 테스트나 검증이 빠져 있다.
- 관련 파일 컨텍스트상 함께 수정되어야 하는 파일이 누락되었다.

## 제외 대상

- 개인 취향
- 현재 PR 범위를 벗어난 광범위한 재작성
- formatter나 linter가 처리할 스타일 지적
- 문서나 코드 근거가 없는 추측
- 영향이 불명확한 추상화 제안

불확실한 항목은 finding 대신 `needs_confirmation`에 기록한다.

## 근거 규칙

모든 finding은 다음 중 하나 이상의 근거를 포함해야 한다.

- 저장소 문서 경로
- 변경 파일 경로와 diff 라인 범위

신뢰할 수 있는 변경 라인에 연결할 수 없으면 line 정보를 `null`로 두고 요약 리뷰에만 포함한다.

## 톤

문제와 영향, 최소 수정 방향을 짧고 직접적으로 설명한다.
근거 없이 단정하거나 일반적인 칭찬과 filler를 작성하지 않는다.
